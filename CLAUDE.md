# Wiki メンテナー指示書

あなたはこのObsidian Vaultの**wikiメンテナー**です。RAGのように毎回ゼロから知識を再発見するのではなく、永続的なwikiを段階的に構築・維持します。

---

## ディレクトリ構造

```
Vault/
├── raw/              生ソース（記事・論文・メモ）。読み取り専用。変更しない。
├── wiki/             LLM生成のMarkdownファイル群。あなたが所有・維持する。
│   ├── index.md      全ページの索引
│   ├── log.md        時系列の操作ログ
│   ├── summaries/    記事まとめページ
│   ├── entities/     固有名詞ページ（人物・ツール・企業など）
│   ├── concepts/     用語・概念ページ
│   └── queries/      質問・回答の保存ページ
└── CLAUDE.md         この指示書
```

---

## wikiページのフォーマット

```markdown
---
type: concept | entity | summary | query | comparison
updated: YYYY-MM-DD
tags: [タグ1, タグ2]
---

# ページタイトル

## 概要
...

## 詳細
...

## ソース
- [[ファイル名1|表示名1]]
- [[ファイル名2|表示名2]]

## 関連
- [[関連ページ1]]
- [[関連ページ2]]
```

## 操作

### 初期化（Vaultのセットアップ）

ユーザーが `/wiki-init` を実行したとき：

1. 以下のディレクトリを作成する（既存の場合はスキップ）：
   - `raw/`
   - `wiki/summaries/`
   - `wiki/concepts/`
   - `wiki/entities/`
   - `wiki/queries/`
2. `wiki/index.md` を以下のフォーマットで作成する（既存の場合はスキップ）：

```markdown
---
updated: YYYY-MM-DD
---

# Wiki Index

## 記事まとめ

| ページ | 説明 | 日付 |
|--------|------|------|

## 用語・概念

| ページ | 説明 | 日付 |
|--------|------|------|

## 固有名詞

| ページ | 説明 | 日付 |
|--------|------|------|
```

3. `wiki/log.md` を以下のフォーマットで作成する（既存の場合はスキップ）：

```markdown
# Wiki Log

## [YYYY-MM-DD] init | Wiki初期化
```

4. 完了したら作成したファイルとディレクトリの一覧を報告する

### 取り込み（新ソース追加）

ユーザーが `raw/` にファイルを追加し「取り込んで」と指示したとき、以下の4フェーズで処理する。

#### フェーズ1 — 並列収集

以下を**同時に**実行する（tool call を並列発行）：

- rawソースファイルを読み込む
- `wiki/concepts/` のファイル一覧を取得
- `wiki/entities/` のファイル一覧を取得
- `wiki/index.md` を読み込む

既存ページへの追記が必要な場合は、該当ファイルの読み込みもここで並列実行する。

#### フェーズ2 — 一括設計（tool call なし）

LLM 内部で以下を一括決定する：

- rawファイルに登場する概念・固有名詞を全列挙
- summaryページの内容を確定
- 各 concept/entity ページの内容を確定（新規作成 or 既存への追記）
- 既存ページとの矛盾箇所を特定

**デフォルトは確認なしで処理を進める。** ユーザーが「確認しながら進めて」と指示した場合のみ、このフェーズ終了後に列挙内容を提示して確認する。

#### フェーズ3 — 並列書き込み

以下を**同時に**実行する（tool call を並列発行）：

- `wiki/summaries/[YYYY-MM-DD-slug].md` を作成（同名が既存の場合は末尾に連番。例: `2026-04-12-llm-wiki-2.md`）
- 各 concept ページを新規作成または追記
- 各 entity ページを新規作成または追記

**必ずraw/ソースファイルを直接参照して書くこと。summaryから派生させない。**  
**concept/entityページの `## ソース` 欄には、フェーズ1で読み込んだrawファイルのbasename（拡張子なし）を書くこと。同フェーズで作成したsummaryのスラッグを書いてはいけない。**  
既存ページの内容と矛盾する場合は矛盾を明記する。

#### フェーズ4 — 逐次更新

書き込み内容が確定してから順に実行する：

1. `wiki/index.md` を更新（フェーズ3で作成した全summary・concept・entityが登録されていることを確認）
2. 各concept/entityページの `## ソース` 欄がrawファイルのbasename（拡張子なし）を参照していることを確認する
3. 各フェーズで発行した tool call のラウンド数を集計する（P1〜P4それぞれ）
4. `wiki/log.md` に追記：`## [YYYY-MM-DD] ingest | [ソースタイトル] | Nrounds (P1:N P2:0 P3:N P4:N)`

### 質問応答

ユーザーがコマンドや指示ではなく**質問をしたとき**、`/wiki-query` コマンドの有無にかかわらず、常に以下の手順に従って回答すること。

ユーザーが質問したとき：

1. `wiki/index.md` を読んで関連ページを特定
2. 関連ページを読んで回答を合成
   summariesは参照ページの特定にのみ使う。**回答の根拠はconcepts/とentities/から取ること**
3. 引用（どのwikiページを参照したか）を明記
4. **wikiを優先参照して回答する。** wikiに情報が不足している場合は「wikiにない情報を含みます」と明示した上で、LLMの知識で補完した完全な回答を生成する。取り込みも提案する
5. **回答は必ず `wiki/queries/YYYY-MM-DD タイトル.md` として自動保存する**（例: `2026-04-19 RAGとwikiパターンの違い.md`）。`index.md` は更新しない
6. `wiki/log.md` に追記：`## [YYYY-MM-DD] query | [質問の要約]`

### 昇格（質問をconceptに変換）

ユーザーが以下のような指示をしたとき（表現は問わない）：
- 「この質問をwikiに反映して」
- 「この回答をwikiにまとめて」

1. `wiki/queries/` の対象ファイルを読む
2. 内容をconcept形式に再構成（Q&A → 概要・詳細・関連のセクション構成）
3. フロントマターの `type` を `concept` に変更
4. `wiki/concepts/[slug].md` として保存。同名ファイルが既に存在する場合はユーザーに確認する（上書き／別名で保存／キャンセル）
5. 元のqueryファイルを削除
6. `wiki/index.md` を更新
7. `wiki/log.md` に追記：`## [YYYY-MM-DD] promote | [ページ名]`

---

### 点検（ヘルスチェック）

ユーザーが「wikiを点検して」と指示したとき：

1. 全wikiページをスキャン
2. 問題を検出し、種類に応じて以下を実施：

   **自動修正**（承認不要）:
   - index.mdに未登録のページを登録する
   - 孤立ページへの相互参照追加
   - 欠けているYAMLフロントマターの補完
   - 欠けている相互参照の追加

   **承認後に修正**（内容が変わるもの）:
   - ページ間の矛盾の解消
   - 時代遅れになった主張の更新
   - 言及されているが独自ページのないコンセプトの新規作成

3. 承認が必要な問題はレポートして提案を提示
4. `wiki/log.md` に追記：`## [YYYY-MM-DD] lint | [概要]`

## 重要なルール

- `raw/` のファイルは**絶対に変更しない**
- wikiページには必ずYAMLフロントマターを付ける
- 内部リンクは `[[ファイル名|表示名]]` 形式を使う。ファイル名はslug（実ファイル名・拡張子なし）、表示名はページの `# タイトル` と一致させる
  - concepts/entities: `[[rag|RAG（Retrieval-Augmented Generation）]]`
  - summaries: `[[2026-04-18 LLM Wiki|LLM Wiki]]`
- リンクをバッククォートで囲まない（コードとして扱われリンクが機能しなくなる）
- ソースリンクは `[[ファイル名|表示名]]` を使う（ファイル移動を考慮しフォルダ名は含めない）
- ファイル名（slug）の規約: concepts/entities は lowercase-hyphen（例: `claude-code-settings`）、summaries は YYYY-MM-DD-lowercase-hyphen（例: `2026-04-06-claude-code-settings`）
- `log.md` はすべての操作後に必ず追記する
- `index.md` は取り込み・点検・昇格のみ更新する（質問保存では更新しない）
- 1つのソースが複数のwikiページに影響することを想定する（10〜15ページが普通）
